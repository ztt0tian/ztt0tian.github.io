提供高质量的上下文

高质量上下文通常包含：

  1. 问题背景

    • 说明你要做什么、为什么做、目标是什么。
    • 例：你在做量化交易平台，想写一个 EMA 金叉死叉策略检测模块。

  2. 技术栈和环境

    • 指明语言、框架、库、版本等。
    • 例：Java 17 + Spring Boot + MyBatis Plus + RabbitMQ。

  3. 现有代码或结构

    • 提供类、方法、数据结构、表结构示例，让 AI 知道你已有的实现。
    • 例：你有一个 MarketDataEvent DTO，包含 BigDecimal price、LocalDateTime timestamp。

  4. 目标和输出

    • 明确希望生成的内容或行为。
    • 例：生成一个监听实时行情，计算 EMA7/14 并判断金叉/死叉的服务类。

  5. 限制和约束

    • 精度要求、性能要求、风格约束等。
    • 例：所有价格用 BigDecimal，保留 8 位小数；MQ 消费者需要确认机制

    举例说明

    差的上下文：

    “帮我写一个 EMA 金叉死叉的函数”

    问题：

    • 没有说明语言
    • 没有说明 EMA 的周期
    • 没有说明数据来源或数据结构
    • 没有说明输出要求

    结果：AI 可能生成一个 Python 函数或者伪代码，可能与你的项目环境不兼容。

    高质量上下文：

    “我在做一个 Java 17 + Spring Boot 的量化交易平台。
    我有一个 DTO MarketDataEvent，里面有字段 BigDecimal price 和 LocalDateTime timestamp。
    我希望写一个 Spring 服务类 EmaStrategyService，每次收到 MarketDataEvent 都计算 EMA7 和 EMA14，并判断金叉或死叉。
    输出信号要封装成 AlertSignal 对象，包含 signalType、price、timestamp，并发送到 RabbitMQ 队列 alert.signal.queue。
    计算必须使用 BigDecimal，保留 8 位小数，确保高精度。”

    分析：

    • 完整说明了语言、环境、数据结构、目标、输出和约束
    • AI 可以直接生成贴合你项目的 Java 代码，包括 MQ 发送逻辑和精度处理

    💡 总结公式：

    高质量上下文 = 背景 + 技术栈 + 数据/结构 + 目标输出 + 限制约束

小步迭代

1️⃣ 核心含义

  • 小步：每次迭代只做有限的、明确的功能或改进，不试图一次性实现所有功能。

  • 迭代：完成一小步后进行测试、反馈、调整，然后继续下一步。

  • 快速反馈：每次小步完成后就可以验证结果是否正确，避免大规模错误积累。

  • 持续改进：通过多次小步迭代逐渐完善系统，降低风险，提高质量。

  • 举例说明

    大而笼统的做法（非小步迭代）：

    想一次性实现完整量化交易平台：

    • 数据采集
    • 策略分析
    • 模拟交易
    • 回测
    • 前端可视化
    • 部署运维

    问题：

    • 工期长
    • 容易出错
    • 很难快速看到成果

    小步迭代做法：

    1. 迭代 1：只实现 BTC 日 K 线数据采集，并保存到 PostgreSQL。
    2. 迭代 2：在迭代 1 基础上，实现 EMA7/14 指标计算，并生成 MarketDataEvent。
    3. 迭代 3：添加简单策略分析，生成金叉/死叉信号。
    4. 迭代 4：实现模拟交易系统，记录 MockOrder。
    5. 迭代 5：加入前端可视化，展示 K 线和策略信号。

    好处：

    • 每个迭代周期都能交付可运行、可验证的功能
    • 可以早期发现逻辑或技术问题
    • 方便调整策略或架构设计
    • 减少大规模返工

    💡 一句话总结：

    小步迭代就是“把大山切成小块，每次搬一块，搬完检查，再搬下一块”，避免一次搬整个大山造成崩塌。

明确输入、输出和边界情况

“明确输入、输出和边界情况”是软件开发中一个非常实用的原则，尤其在函数、服务、接口或者算法设计时。它能帮助你定义清晰的职责、避免模糊和潜在 bug。

1️⃣ 核心含义

  1. 输入

    （Input）

    • 明确函数或服务接收什么数据
    • 类型、格式、取值范围
    • 例如：MarketDataEvent，BigDecimal price，时间戳 LocalDateTime timestamp

  2. 输出

    （Output）

    • 明确函数或服务返回什么结果
    • 类型、格式、取值范围
    • 例如：AlertSignal 对象，包含 signalType、price、timestamp

  3. 边界情况

    （Boundary Conditions / Edge Cases）

    • 定义在特殊、极端、异常输入下的行为
    • 避免程序崩溃或逻辑错误
    • 例如：
      • EMA 计算时，历史价格数据不足 7 条或 14 条
      • 数据为空或 null
      • 价格为 0 或负数（异常数据）
      • MQ 消息发送失败或超时

    2. 当前价格为 null → 抛出异常或记录日志，跳过计算
    3. EMA 计算结果出现非数字（NaN） → 校验并忽略该条数据
    4. MQ 队列不可用 → 重试机制或缓存待发送

3️⃣ 为什么重要

  • 避免逻辑错误：很多 bug 来源于没有处理边界情况
  • 可测试性强：输入、输出、边界都明确后可以写单元测试覆盖各种场景
  • 提高可维护性：别人看到接口定义就知道使用方法和限制
  • 与小步迭代结合：每个迭代的小功能都可以定义好输入输出和边界，快速验证

💡 总结公式：

明确输入 + 明确输出 + 明确边界 = 函数/服务清晰职责 + 易测试 + 稳定可靠

毫不留情的审查代码

高效审查方法

(1) 重点审查

  • 核心逻辑：策略算法、交易下单逻辑、资金计算、止盈止损逻辑
  • 边界条件：空数据、零值、负值、极端行情
  • 外部交互：数据库、MQ、REST API 调用
  • 精度敏感代码：金额、百分比、指标计算

对于普通工具类或简单 DTO，可以只做快速扫描，不必逐行审查。

(2) 批量/模块化审查

  • 把代码按模块分块，比如：数据采集、策略分析、下单执行
  • 每块单独审查核心逻辑，模块内部非核心代码可以轻量检查

(3) 自动化辅助

  • 静态分析工具：Checkstyle、SonarQube、SpotBugs 自动检查代码风格和潜在 bug
  • 单元测试：覆盖边界条件，快速验证逻辑正确性
  • 集成测试：模拟真实数据，验证服务间交互是否稳健

自动化 + 单元测试可以大幅降低人工逐行检查负担。

(4) 审查清单法

  • 列出关键审查点，而不是每行代码：
    • 输入验证是否到位
    • 输出类型和精度是否正确
    • 异常处理是否健全
    • 性能热点是否优化
  • 遇到 AI 生成的新方法，按清单快速对照检查
  • 对于不在清单范围的辅助代码，可随机抽样或快速浏览

(5) 小步迭代 + 及时反馈

  • 每次 AI 生成的代码只提交一小块功能
  • 立刻运行单元测试、模拟交易回测
  • 及时发现问题并反馈给 AI，修改成本低

2️⃣ 总结

毫不留情的审查 ≠ 人工逐行盯着看

  • 核心逻辑重点盯
  • 非核心代码快速扫
  • 自动化工具和单元测试辅助
  • 小步迭代降低风险

自动要求生成测试

核心含义

当 AI 生成业务逻辑或功能代码时，同时要求它 生成对应的单元测试或集成测试，确保功能可验证、可回归，而不是只给出逻辑实现。

换句话说，你不仅要 AI 帮你写“功能代码”，还要它帮你写验证代码，形成闭环。

1️⃣ 为什么重要

  1. 验证正确性
    • AI 生成的代码可能有逻辑漏洞或边界遗漏
    • 测试可以快速验证核心功能是否符合预期
  2. 降低人工审查成本
    • 有了测试，部分“毫不留情的审查”可以自动化：运行测试就能发现问题
  3. 支持小步迭代
    • 每次迭代功能生成，自动生成测试
    • 小步交付 + 测试覆盖 → 快速反馈循环
  4. 提升可维护性
    • 后续修改代码可以直接跑测试，保证原逻辑不被破坏

2️⃣ 举例说明

场景：生成 EMA 金叉/死叉服务

  • 功能请求：

    “帮我生成一个 Java 类 EmaStrategyService，计算 EMA7/EMA14 金叉死叉，并输出 AlertSignal。”

  • 自动生成测试请求：

    “同时帮我生成 JUnit 测试类 EmaStrategyServiceTest，覆盖以下情况：

    1. 正常历史数据产生金叉信号
    2. 历史数据不足 14 条 → 不生成信号
    3. 当前价格为 null → 抛出异常
    4. 连续两次金叉 → 去重处理正确
      测试用例要使用 BigDecimal 保证精度。”

这样，生成的 AI 代码不仅有业务逻辑，还自带验证逻辑，你可以直接运行测试，发现潜在问题。

3️⃣ 在提示词里的用法

在 Vibe-Coding 或 AI 提示词里，可以这样表述：

核心是 “同时生成测试”，不是事后再手动写。

💡 一句话总结：

“自动要求生成测试” = 每次 AI 输出功能代码时，自动生成覆盖边界、异常和核心逻辑的验证测试，形成自带验证的交付物。
