背景

我现在准备做一个基于OKX的量化交易平台，我自身是一名java语言的后端开发，我希望你做出来的程序清晰易懂。你不必一次性完成所有功能，大致的框架可以先搭出来，我希望你渐进式的完成特定功能，完成一个由我来验证是否正常。

功能模块

大致分为以下功能模块

  • 数据采集
  • 告警分析与处理
  • 仓位管理

数据采集模块

  • 采集所有USDT本位永续合约币种保存入库（t_instrument）,从启动开始每24小时采集一次更新，设置一个标识位：是否采集（is_collect）,默认不采集，为便于验证首先让BTC-USDT-SWAP默认是采集的。

  • 针对采集的USDT本位永续合约进行日k、周k、月k维度下的历史k线原始数据，保存入库（t_k_line），记得格式话k线开盘时间，首次只需要采集近30个已完结周期的数据。后续再增量采集上一周期的数据（每日零点拉昨天的日k，每周一零点再拉上周的周k，每月1号零点再拉上月的月k）。

  • 计算近10个日k、周k和月k的boll(2,20)指标，保存入库（t_boll）,关于boll的指标需要计算这些值：

    中轨，上轨，下轨，布林极限%BB，当前周期的布林带宽度，前n个周期中轨的线性回归斜率角度值，n的具体值，n支持在apollo中可配置（boll_pre_n_count默认5）

    ​ 对5个离散点进行线性回归，拟合出一条直线，求出其斜率，再换算为角度。

    线性回归斜率角度的计算逻辑
    输入数据
    获取最近连续5个周期的BOLL中轨数值：
    [M1, M2, M3, M4, M5]
    对应时间顺序从旧到新（M1最旧，M5最新）。

    数据映射
    将这5个点视为平面坐标：

    X轴：周期序号 [1, 2, 3, 4, 5]

    Y轴：中轨数值 [M1, M2, M3, M4, M5]

    线性拟合求斜率
    使用最小二乘法计算拟合直线的斜率 k：

    text
    n = 5
    sum_x = 1+2+3+4+5 = 15
    sum_y = M1+M2+M3+M4+M5
    sum_xy = 1M1 + 2M2 + 3M3 + 4M4 + 5*M5
    sum_xx = 1^2+2^2+3^2+4^2+5^2 = 55

    k = (nsum_xy - sum_xsum_y) / (nsum_xx - sum_xsum_x)
    斜率转角度（弧度 → 度）
    注意：这里角度是价格-时间坐标系下的视觉角度，需考虑 价格与时间的缩放比。

    如果不缩放（X轴1周期=1单位，Y轴1价格=1单位），直接计算：
    角度 = arctan(k) * (180/π)

    更符合实际K线图：需要 价格单位与时间单位的视觉比例，常见做法是引入 Y轴每单位价格对应的像素 / X轴每周期对应的像素 比例 ratio（例如1周期占10像素，1价格占5像素，则 ratio=0.5）。
    调整后的斜率：k_adj = k * ratio
    角度：angle = arctan(k_adj) * (180/π)

    输出结果
    角度范围通常为 -90° 到 +90°：

    正值 → 中轨向上倾斜（多头趋势增强）

    负值 → 中轨向下倾斜（空头趋势增强）

    接近0 → 中轨走平（震荡）

    示例（无缩放比例，纯数值计算）
    假设最近5个中轨值：[100, 102, 104, 106, 108]

    计算得 k = 2.0

    angle = arctan(2.0) * 180/π ≈ 63.4°

    实际K线图中Y轴价格波动范围远小于X轴周期跨度，通常角度在 -45°~45° 之间。强烈建议引入视觉比例，否则角度会明显偏大。

  • 利用websocket采集实时行情数据，并依赖历史数据计算出实时boll指标

    这里需要考虑性能，需要尽可能快的知道实时周k或月k维度下的实时boll指标。（你可以考虑将用于计算实时boll中固定的那部分历史k线数据从数据库查出来并缓存下来，计算实时指标时直接从缓存拿进行计算，如果数据有缺漏那还是要通过接口获取并更新缓存 ），日志打印出来实时指标结果。

  • 当前币对价格触及boll的上轨或者下轨（前提时需要存在上轨或者下轨）时需要出发告警通知消息，推送至mq的告警消息队列中，消息中需要记录：boll指标信息、币种价格信息、币种价格相对触发告警的boll上轨或下轨的相对偏移百分比、以及boll上轨和下轨的百分比差距、近期中轨斜率角度

告警分析与处理模块

  • 负责监听告警通知消息，并通过钉钉推送消息（同一合约同一天同维度的同种告警一天最多推送一次钉钉通知消息）

  • 可能存在需要推送计划下单通知消息，这依赖具体的计划下单策略，需满足以下条件

     触及或跌穿周k维度或月k维度boll下轨

下单及仓位管理模块

  • 负责监听并处理计划下单通知消息
  • 每5min同步当前账号的持仓信息至数据库中
  • 下单限制

​ （1）下单USDT为账号可用余额的10%

​ （2）下单倍数：非ETH、BTC的合约均设置为2倍（apollo可配置），ETH和BTC的合约设置为10倍（apollo可配置），下单价格即为合约的实时价格。

​ （3）下单仓位模式：逐仓，注意一定不要全仓

​ （4）若已有持仓或者近30天已经有同币种合约下单的话，则放弃下单，并推送钉钉消息告知放弃下单原因。

​ （5）不设止损位置，仅设置止盈位置为boll的上轨价格

​ （6）非ETH、BTC的合约实时价格需要低于boll的下轨20%（apollo可配置）才能自动下单

​ （7）近期的中轨斜率不能小于-30度

  • 下单成功需要推送钉钉消息
